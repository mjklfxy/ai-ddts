# 数据采集方案对比分析

## 背景

当前系统需要从吉客云（JackYun）获取订单数据。吉客云标准 OpenAPI（`wms.order.query-info.page.v2`）可以返回订单号、SKU、仓库等基础字段，但部分字段（收件人、收货地址、电话等）API 可能返回空值。因此需要额外的数据采集手段补全这些缺失字段，保证推送文件和异常订单明细的完整性。

---

## 方案一：PyAutoGUI 模拟键鼠操作桌面 App（当前方案）

### 实现方式

通过 Python 的 `pyautogui` + `pygetwindow` + `uiautomation` 库，自动操作吉客云 OMS 桌面客户端（Flutter 应用），执行"登录 → 筛选订单 → 导出 xlsx"的完整人工操作流程：

- **窗口定位**：通过 `pygetwindow` 按标题查找、或 `uiautomation` 按 ClassName 查找吉客云窗口，自动激活并置顶
- **操作执行**：模拟鼠标点击、悬停等
- **坐标操作**：基于相对坐标进行点击，不依赖屏幕分辨率和 DPI 缩放
- **数据回填**：导出的 xlsx 文件被读取为收件人/地址/电话兜底数据，当 API 返回空值时从 xlsx 中补全

### 方案一优势

- **合法合规**：操作的是公司已授权的吉客云桌面客户端，正常使用已购买的软件功能
- **数据完整**：能拿到 API 不返回的收件人姓名、详细地址、手机号等字段
- **无接口依赖**：不受 OpenAPI 字段覆盖范围限制，桌面端能看到的都能导出
- **可全自动**：全程无需人工介入，支持定时循环执行
- **坐标容错**：使用窗口相对坐标或图像识别，不因窗口位置/分辨率变化而失效

### 方案一劣势

- **环境依赖重**：需要 Windows 环境、吉客云客户端已安装并保持登录状态
- **速度较慢**：每个步骤需要 `delay` 等待（点击后页面响应、导出文件生成等），单次导出约 30-60 秒，但是本项目对时间要求不高
- **稳定性受 UI 影响**：吉客云客户端界面改版时，截图和坐标需要重新校准
- **独占桌面**：运行时占用鼠标键盘，不能同时做其他操作
- **调试成本较高**：定位失败时需要重新截图或调整坐标

## 方案二：Codex 用户增强类的ai代理方法

### 实现方式

通过 AI 代理（如 OpenAI Codex 或 Claude Agent）驱动浏览器自动化（Playwright/Selenium），自动登录吉客云网页后台、导航到订单管理页面、解析页面 DOM 并提取数据。

### 方案二优势

- **理论上可覆盖任意页面**：不依赖 API，能采集网页端所有可见数据
- **灵活性高**：页面字段变更时可通过调整 prompt 适应，无需修改代码逻辑

### 方案二劣势

- **前置条件繁多**：需要搭建浏览器自动化基础设施、配置 Playwright/Selenium 环境、管理 Cookie 和登录态生命周期、处理验证码和 MFA 认证
- **迁移极不稳定**：吉客云网页端没有官方文档保证 DOM 结构稳定，页面改版频繁，每次改动都可能导致 AI 代理定位失败，排查和恢复成本远高于固定坐标/截图方案
- **速度慢**：页面加载、渲染、AI 推理识别、元素等待等耗时远超桌面 App 直接操作
- **资源消耗大**：需要运行完整浏览器实例 + AI 模型推理，内存和 CPU 开销高
- **错误率较高**：网络波动、页面加载超时、DOM 元素动态变化、AI 误识别等异常场景叠加
- **合规灰色地带**：自动化操作浏览器可能违反平台服务条款

## 方案三：Win32 UI Automation / uiautomation 辅助功能 API

### 实现方式

通过 Windows 原生辅助功能框架（`UIAutomation` / `MSAA`），以 `uiautomation` 或 `pywinauto` 库按控件类型、AutomationId、Name 等属性精确定位窗口内的 UI 元素，直接读取文本值或触发点击，无需坐标。

### 方案三优势

- **不依赖坐标**：按控件属性定位，窗口位置和分辨率变化不影响
- **可读取控件文本**：直接获取控件内容，无需截图解析
- **原生 Windows 支持**：无需额外安装大型依赖

### 方案三劣势

- **对 Flutter 应用完全无效**：吉客云桌面客户端为 Flutter 构建，整个界面用 Skia 引擎渲染在一块 Canvas 上，所有"按钮"和"文字"均为画布像素，Windows 辅助功能树中只有一个不可穿透的顶层 FlutterWindow 节点，无任何子控件可定位
- **无法读取 Canvas 内文本**：Inspect.exe / Accessibility Insights 工具无法识别 Flutter 控件树内的任何元素
- **适用范围窄**：仅适合 WinForms、WPF、MFC 等传统 Win32 控件体系的应用

---

## 方案四：OCR 截图识别

### 实现方式

截取吉客云窗口区域截图，使用 OCR 引擎（PaddleOCR、Tesseract、Windows.Media.Ocr）识别截图中的文字内容，配合坐标或区域裁剪提取目标字段，再进行鼠标点击操作。与方案一的区别在于：方案一依赖固定坐标点击，方案五依赖识别出的文字位置驱动后续操作。

### 方案四优势

- **不依赖固定坐标**：通过识别文字内容定位，界面布局轻微变化时自适应
- **可验证操作结果**：截图后用 OCR 确认页面状态，再决定下一步操作，容错率更高
- **无控件树依赖**：Canvas 渲染界面同样可用，只要文字可见即可识别

### 方案四劣势

- **识别精度受字体和 DPI 影响**：吉客云界面字体较小、DPI 缩放时识别率下降，需要额外预处理（放大、灰度、二值化）
- **速度慢于纯坐标方案**：每步操作前需截图 + OCR 推理，比直接 `pyautogui.click(x, y)` 慢。
- **引入大型依赖**：PaddleOCR 模型文件约 100MB+，部署成本高；Tesseract 中文识别率对复杂布局较差
- **本质仍属桌面自动化**：仍需保持 Windows 环境和客户端登录状态，独占桌面的根本问题未解决

---

## 方案五：网络抓包逆向吉客云私有 API

### 实现方式

使用 Fiddler / mitmproxy / Charles 等代理工具拦截吉客云桌面客户端发出的 HTTPS 请求，分析其接口协议、签名算法和请求参数，再用 Python 直接构造相同请求调用私有 API，绕过桌面端直接获取数据。

### 方案五优势

- **速度极快**：直接 HTTP 请求，无 UI 操作等待，获取一页数据约 1-3 秒
- **不占用桌面**：无需保持桌面客户端运行，可在无界面服务器环境执行
- **数据最完整**：私有 API 返回的字段通常比 OpenAPI 更全，桌面端展示什么就能拿到什么
- **稳定性高**：接口协议比 UI 更稳定，版本升级频率远低于界面改版

### 方案五劣势

- **存在法律和合规风险**：逆向分析私有接口协议未经吉客云授权，违反软件使用条款；商业场景下需法务评估
- **抓包有难度**：桌面客户端通常配置 SSL Pinning 或使用 Flutter 内置 TLS，需要额外手段（如 Frida hook、系统代理注入）才能解密 HTTPS 流量
- **接口不稳定**：吉客云有权随时修改私有接口结构、鉴权参数或签名算法，无通知无文档，维护成本高
- **鉴权机制复杂**：私有接口通常使用动态 token、时间戳签名或设备指纹。

---

## 结论

**采用方案：PyAutoGUI 模拟键鼠操作桌面 App ，作为当前实施方案**，理由如下：

1. **Codex 用户增强方案前置条件过多**：需要搭建浏览器自动化基础设施、维护登录态和 Cookie 生命周期、处理反自动化检测。吉客云网页端 DOM 结构无官方稳定性承诺，改版频繁，方案迁移和适配极不稳定，不适合作为核心数据链路的依赖。

2. **传统爬虫方案不合法**：商业场景下未经授权抓取平台数据存在明确的法律风险，不符合公司合规要求。

3. **PyAutoGUI 方案在当前场景下是最优解**：吉客云桌面客户端是公司正常购买使用的软件，自动化操作属于正当使用范畴。通过窗口相对坐标 + 图像识别的双重定位机制，解决了标准 API 无法返回部分字段（收件人、地址、电话）的问题。架构上通过 OpenAPI 获取基础订单数据 + PyAutoGUI 导出 xlsx 兜底补全，兼顾了速度（大部分订单走 API），完整性（缺失字段走 xlsx 兜底）和安全性，是当前条件下唯一同时满足合规、稳定、完整三项要求的技术方案。
